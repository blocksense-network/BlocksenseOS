    the configuration, service identity + signed responses, the ZK

** Execution policy

   - **Upstream-first.** Anything a different confidential-computing
     product would also need (TEE backends, measurement tooling, image
     profiles, verifier machinery, provisioning protocol) is
     contributed to ReproOS / reprobuild, never re-implemented here.
     From M0 onward this repo accepts no new platform-level TEE code.
   - **Gate on upstream milestones explicitly.** Each milestone below
     names the ReproOS-Remote-Attestation campaign milestones it
     consumes. If upstream slips, this campaign waits rather than
     forking.
   - **No milestone is done without its verification gate green**, in
     CI wherever hardware permits (mock + TPM tiers in plain CI; CVM
     gates on real hosts, with recorded-evidence fixtures for CI).
   - **No false security claims during the interim.** README/SECURITY
     text must describe mocked capabilities as mocked until the gate
     that makes them real is green (the v0.1.0 docs failed this test;
     v0.2.0 docs fix it and this rule keeps it fixed).

** M0: Scope lock, upstream gates, prototype freeze
   :PROPERTIES:
   :status: planned
   :upstream: none (coordination only)
   :END:

*** Problem
    The prototype invites incremental "make the mock real" work in
    the wrong repo. Lock the new division of responsibilities before
    any code moves.

*** Deliverables
    - [ ] Decision record adopting the disposition table above
      (reviewed by Blocksense + Metacraft platform maintainers)
    - [ ] Contribution path agreed with the ReproOS team: which
      prototype learnings (agent API shape, report JSON schema,
      Justfile/CI patterns) feed upstream design, and who carries them
    - [ ] Freeze notice in README + SECURITY.md: platform security
      features are delivered by the upstream campaign; interim mock
      status stated plainly
    - [ ] CI: mark the =tee_type= matrix jobs as exercising mocks
      (rename, so green ≠ "TEE works")

*** Verification
    - Decision record merged; README/SECURITY contain no claim that a
      mocked capability is real (checked by doc review, recorded
      here).

** M1: Blocksense services as reprobuild recipes
   :PROPERTIES:
   :status: planned
   :upstream: none (uses production reprobuild package model)
   :END:

*** Problem
    The services build only via the Nix flake. The ReproOS
    configuration (M2) needs them as native reprobuild recipes.

*** Deliverables
    - [ ] =repro.nim= recipes for cpp-echo, rust-echo, rust-client
      (typed build actions; pinned toolchains; deterministic outputs)
    - [ ] Recipe CI alongside the existing flake CI (parity phase)
    - [ ] Reproducibility gate: build-twice byte-equality for each
      service binary

*** Verification
    - t_services_repro_build: all three recipes build green in CI.
    - t_services_deterministic: byte-identical across two builds.
    - t_echo_parity: recipe-built binaries pass the existing echo
      integration tests (same Ruby harness).
